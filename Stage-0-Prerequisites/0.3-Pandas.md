## Key Takeaways — Read This Before Any Project

1. **Always run these 4 lines first on any dataset:** `df.head()` → `df.info()` → `df.describe()` → `df.isnull().sum()`
2. **Fill missing values with group median, not global** — `groupby().transform(lambda x: x.fillna(x.median()))`
3. **Use `groupby().transform()` not `groupby().agg()`** when you need group stats as a new column on every row
4. **Your model never sees a DataFrame** — always end with `.to_numpy()` or `.values`
5. **`map(dict)` is how you convert categories to numbers** — the most common encoding for ML
6. **`loc` = by name, `iloc` = by position** — never mix them

### `pd.DataFrame` and `pd.Series` ⭐⭐⭐ 🧠

| | |
|---|---|
| **What** | DataFrame = a table (rows + columns). Series = a single column. |
| **Why** | Raw data comes as CSV, JSON, SQL, Excel — you need a structure to hold and work with it in Python. |
| **When** | The moment you load any dataset. Always. |
| **Where** | Start of every ML pipeline — before inspection, cleaning, or modeling. |
| **Who** | Every ML engineer, data scientist, AI engineer uses this daily. |
| **How** | Create from a dictionary, a file, or a database query. |
| **How often** | Every single project without exception. |
| **How much** | Know what it is and how to create one. Details come with practice. |

```python
import pandas as pd
import numpy as np

# DataFrame — from a dict (most common way to create manually)
df = pd.DataFrame({
    'name':   ['Alice', 'Bob', 'Carol', 'Dave'],
    'age':    [25, 30, 35, 28],
    'salary': [70000, 85000, 90000, 75000],
    'dept':   ['Eng', 'Sales', 'Eng', 'HR']
})

print(df.shape)    # (4, 4) — rows, columns
print(df.dtypes)   # column data types
print(df.columns)  # column names

# Series — just one column
s = df['age']
print(type(s))     # <class 'pandas.core.series.Series'>
print(type(df))    # <class 'pandas.core.frame.DataFrame'>
```

> **Real scenario:** You download a Kaggle dataset → `pd.read_csv()` → it becomes a DataFrame → you clean it → `.to_numpy()` → model trains on it. Every ML project follows this exact path.

### `pd.read_csv()` ⭐⭐⭐ 🧠 Pattern, 📖 Options

| | |
|---|---|
| **What** | Reads a CSV file from disk or URL into a DataFrame. |
| **Why** | 80% of ML datasets are CSV files. This is the most common data loading operation in AI. |
| **When** | First line of almost every ML notebook. |
| **Where** | Data ingestion — before any cleaning or modeling. |
| **Who** | Everyone. Always. |
| **How** | Point it at a file path. It handles headers, delimiters, and types automatically. |
| **How often** | Every project. |
| **How much** | Know the common options below. Don't memorize all of them — look them up. |

```python
# Basic
df = pd.read_csv('data.csv')

# Options you'll actually need in real projects
df = pd.read_csv('data.csv',
    usecols=['age', 'salary', 'dept'],   # only load what you need — saves memory
    dtype={'age': int, 'salary': float}, # force column types upfront
    parse_dates=['signup_date'],          # auto-convert date strings to datetime
    na_values=['N/A', '-', '?'],          # treat these strings as NaN
    nrows=5000                            # load only first 5000 rows for testing
)

# Save back
df.to_csv('cleaned.csv', index=False)    # index=False — don't write row numbers as extra column
```

> **Common mistake:** Forgetting `index=False` when saving. Without it, pandas writes an extra `0,1,2,3...` column into your CSV. Every ML engineer gets burned by this at least once.

> **Used in Stage 1:** Loading the Titanic or House Prices dataset before your first classifier.

## SECTION 3 — Inspecting a Dataset

---

### `head()`, `info()`, `describe()`, `isnull().sum()` ⭐⭐⭐ 🧠 Always run all four

| | |
|---|---|
| **What** | Quick inspection tools — show you what your data looks like before touching it. |
| **Why** | Blindly training a model on dirty data produces wrong results silently. You will never know why the model is bad unless you looked at the data first. |
| **When** | Immediately after loading any dataset. No exceptions. |
| **Where** | EDA (Exploratory Data Analysis) — the mandatory first step of every ML project. |
| **Who** | Every ML engineer. Senior engineers do this even more carefully than juniors. |
| **How** | Four lines, always in this order. |
| **How often** | Every project, every new dataset. Non-negotiable. |
| **How much** | Memorize these four. They take 30 seconds and save hours. |

```python
df.head()                          # first 5 rows — visual sanity check
df.info()                          # dtypes + non-null counts + memory — look for wrong types
df.describe()                      # mean, std, min, max — look for impossible values (age=999)
df.isnull().sum()                  # missing values per column — decide fill vs drop strategy

# Additional useful ones
df['dept'].value_counts()          # for text columns — what categories exist, how frequent
df.isnull().sum() / len(df)        # percentage missing — helps decide fill vs drop
df.duplicated().sum()              # duplicate rows — always check before training
df.nunique()                       # number of unique values per column
```

> **Real scenario:** You load a salary dataset. `df.info()` shows `salary` dtype is `object` (string) — someone put "N/A" in the column. `df.describe()` shows max age is 999 — clearly a data entry error. You caught both before training anything. This is exactly why you always inspect first.

> **Used in every stage:** This is the first thing you do in every project from Stage 1 to Stage 10.

```python
df['age']                   # one column → Series
df[['age', 'salary']]       # multiple columns → DataFrame

# Final step before model — select feature columns
X = df[['age', 'salary', 'years_exp']].to_numpy()
```

---

### `df.loc[]` — Label-Based Selection ⭐⭐⭐ 🧠 Pattern, 📖 Syntax

| | |
|---|---|
| **What** | Select rows and columns by their **names**. |
| **Why** | You want to refer to data by what it's called, not where it happens to be positioned. Positions change when you add/remove columns — names don't. |
| **When** | Selecting a named subset, filtering by condition, updating specific cells. |
| **Where** | Data cleaning, feature selection, targeted updates. |
| **How** | `df.loc[row_selector, column_selector]` — both by label. |

```python
df.loc[0, 'name']                            # single cell — row 0, column 'name'
df.loc[0:2, 'name':'salary']                 # range of rows and columns (both inclusive)
df.loc[df['age'] > 28]                       # all rows where condition is True
df.loc[df['age'] > 28, ['name', 'salary']]   # filtered rows + specific columns only
```

---

### `df.iloc[]` — Position-Based Selection ⭐⭐ 🧠 Pattern, 📖 Syntax

| | |
|---|---|
| **What** | Select rows and columns by their **integer position** (0, 1, 2...). |
| **Why** | Sometimes you want "the first 100 rows" or "the last column" regardless of names — like NumPy slicing. |
| **When** | Manual train/test splits, accessing by position when column order is known. |
| **How** | `df.iloc[row_position, col_position]` — both by integer index, like NumPy. |

```python
df.iloc[0]           # first row
df.iloc[0:100]       # first 100 rows
df.iloc[:, 0:3]      # all rows, first 3 columns
df.iloc[:, -1]       # all rows, last column
```

> **Rule:** Use `loc` when you know the name. Use `iloc` when you know the position. Never mix them — it causes silent wrong results.

## SECTION 5 — Filtering Rows

---

### Boolean Filtering ⭐⭐⭐ 🧠

| | |
|---|---|
| **What** | Keep only rows that match a condition — filter by value. |
| **Why** | In ML you regularly need to remove outliers, isolate a class, or create training subsets. |
| **When** | Data cleaning, outlier removal, class-specific analysis, creating subsets. |
| **Where** | Preprocessing — before training. Also in EDA when investigating specific groups. |
| **How** | Write a condition → pandas evaluates True/False for each row → only True rows are kept. |
| **How often** | Multiple times in every project. |

```python
df[df['age'] > 28]                                   # single condition
df[(df['age'] > 25) & (df['dept'] == 'Eng')]          # AND — must use &, not 'and'
df[(df['dept'] == 'Eng') | (df['dept'] == 'Sales')]   # OR — must use |, not 'or'
df[~(df['dept'] == 'HR')]                             # NOT — use ~

df[df['dept'].isin(['Eng', 'Sales'])]                 # cleaner than multiple OR
df[df['salary'].between(70000, 90000)]                # inclusive range filter
```

> **Common mistake:** Using Python's `and`/`or` instead of `&`/`|`. With pandas you must use `&` and `|` for combining conditions. `and`/`or` will raise a ValueError.

---

### `df.query()` ⭐⭐ 📖

| | |
|---|---|
| **What** | Filter using a readable SQL-like string instead of Python syntax. |
| **Why** | Complex conditions with many `&` and `|` become unreadable quickly. `query()` reads like English. |
| **When** | 3+ conditions, or when you want clean readable analysis code. |

```python
df.query("age > 28 and dept == 'Eng'")
df.query("salary > 80000 and dept != 'HR'")

threshold = 80000
df.query("salary > @threshold")    # @ references a Python variable
```

## SECTION 6 — Transforming Data

---

### Adding and Modifying Columns ⭐⭐⭐ 🧠

| | |
|---|---|
| **What** | Create new columns derived from existing ones, or change existing column values. |
| **Why** | Raw data rarely has the features your model needs. You create better inputs — this is called **feature engineering** and it often has the biggest impact on model accuracy. |
| **When** | After cleaning, before training. This is where ML intuition pays off. |
| **Where** | Feature engineering step — middle of every ML pipeline. |
| **How often** | Multiple new columns created in every project. |

```python
df['bonus'] = df['salary'] * 0.10                    # derived column
df['salary'] = df['salary'] * 1.05                   # modify existing

# Conditional column — apply logic per row
df['seniority'] = df['age'].apply(lambda x: 'senior' if x >= 30 else 'junior')

# assign() — chain-friendly, doesn't mutate original, returns new DataFrame
df = df.assign(
    tax = df['salary'] * 0.20,
    net = lambda x: x['salary'] - x['tax']   # can reference column just created above
)

df.rename(columns={'dept': 'department'}, inplace=True)
df.drop(columns=['bonus'], inplace=True)
df.reset_index(drop=True, inplace=True)    # reset row numbers after dropping rows
```

> **Used in Stage 1:** Creating `price_per_sqft` from `price` and `sqft` before training a house price model. The engineered feature often outperforms either raw column alone.

---

### `df.apply()` ⭐⭐⭐ 🧠 Pattern, 📖 Syntax

| | |
|---|---|
| **What** | Apply a custom function to each element of a column, or to each row. |
| **Why** | When no built-in function does exactly what you need. Covers any custom transformation. |
| **When** | Custom feature engineering, multi-column logic per row, text transformation. |
| **How** | On a Series: applies to each element. On a DataFrame with `axis=1`: applies to each row. |

```python
# Per element — simple transformation
df['age_group'] = df['age'].apply(lambda x:
    'young'  if x < 30 else
    'mid'    if x < 45 else
    'senior'
)

# Per row — access multiple columns at once
df['value_score'] = df.apply(
    lambda row: row['salary'] / row['age'] * row['score'],
    axis=1    # axis=1 means "apply across columns for each row"
)
```

---

### `df['col'].map(dict)` ⭐⭐⭐ 🧠

| | |
|---|---|
| **What** | Replace each value in a column using a dictionary lookup. |
| **Why** | ML models only understand numbers. `map()` is how you convert text categories (`'Eng'`, `'Sales'`) into numbers (`1`, `2`). This is called **label encoding**. |
| **When** | Before feeding any categorical column to an ML model. |
| **Where** | Preprocessing step — every classification or regression project. |
| **How often** | Every project that has text/category columns. |

```python
df['dept_code'] = df['dept'].map({'Eng': 1, 'Sales': 2, 'HR': 3})

# Result: 'Eng' → 1, 'Sales' → 2, 'HR' → 3
# Now this column can go into any ML model
```

> **Used in Stage 1:** Converting `gender`, `department`, `city` columns to numbers before your first scikit-learn classifier. Without this, the model cannot run.

## SECTION 7 — Handling Missing Data

---

### `isnull()`, `fillna()`, `dropna()` ⭐⭐⭐ 🧠

| | |
|---|---|
| **What** | Detect, fill, or remove missing values (NaN = Not a Number). |
| **Why** | ML models crash or silently produce wrong results when given NaN. You must handle missing data before training. There is no optional path here. |
| **When** | During data cleaning — after loading, before feature engineering or modeling. |
| **Where** | Preprocessing — mandatory in every pipeline. |
| **How** | Three decisions: is it missing? can you estimate it (fill)? or is too much missing (drop)? |
| **How often** | Real-world datasets almost always have missing values. Always check. |
| **How much** | Know the fill strategies. The choice between them affects model quality. |

```python
# Step 1 — always check first
df.isnull().sum()                    # count missing per column
df.isnull().sum() / len(df)          # percentage missing — guides your strategy

# Fill — when you can estimate the missing value
df['salary'].fillna(df['salary'].mean(), inplace=True)       # fill with mean
df['salary'].fillna(df['salary'].median(), inplace=True)     # median — better for skewed data
df['dept'].fillna('Unknown', inplace=True)                   # categorical → use a placeholder

# BEST PRACTICE — fill with group median, not global
# "What is the typical salary for someone in Engineering?" is better than
# "What is the typical salary for everyone?"
df['salary'] = df.groupby('dept')['salary'].transform(
    lambda x: x.fillna(x.median())
)

# Drop — when too much is missing to fill meaningfully
df.dropna(subset=['salary'])          # drop rows where salary is NaN
df.dropna(thresh=5)                   # keep rows with at least 5 non-null values
```

> **Fill vs Drop decision rule:**
> - < 5% missing → fill with median/mean
> - 5–30% missing → fill using group statistics or a model
> - \> 30% missing → consider dropping the column entirely
> - Target column (y) has missing → always drop those rows

> **Used in Stage 1:** Handling missing `Age` and `Cabin` in the Titanic dataset before your first classifier.

## SECTION 8 — GroupBy

---

### `df.groupby().agg()` ⭐⭐⭐ 🧠

| | |
|---|---|
| **What** | Split data into groups, apply a function to each group, return one row per group. |
| **Why** | You rarely analyze a whole dataset uniformly. You care about patterns within groups — per department, per region, per user segment. |
| **When** | Analyzing group-level patterns in EDA. Comparing model performance across segments. |
| **Where** | EDA step and post-model analysis. |
| **Who** | Every ML engineer and data scientist. |
| **How** | Split by column → aggregate (mean, sum, count, etc.) → one result row per group. |

```python
df.groupby('dept')['salary'].mean()
df.groupby('dept')['salary'].agg(['mean', 'min', 'max', 'count'])

# Multiple groupby keys
df.groupby(['dept', 'seniority'])['salary'].mean()

# Different aggregations per column
df.groupby('dept').agg({
    'salary': ['mean', 'max'],
    'age':    'mean',
    'score':  ['mean', 'std']
})
```

---

### `df.groupby().transform()` ⭐⭐⭐ 🧠 — Critical for Feature Engineering

| | |
|---|---|
| **What** | Like `groupby().agg()` but returns a value for **every row**, not one per group. The output has the same shape as your original DataFrame. |
| **Why** | In ML, you want to add group-level context as a feature on each row. A salary of 90,000 means something different if the dept average is 70,000 vs 110,000. `transform()` brings that context to every row. |
| **When** | Feature engineering — any time you want "what is the X of this row's group" as a column. |
| **Where** | Feature engineering step — one of the most powerful tools in structured ML. |
| **How** | Same as groupby but result shape matches the original DataFrame. |

```python
# groupby().agg() — gives 3 rows (one per dept)
df.groupby('dept')['salary'].mean()

# groupby().transform() — gives 100 rows (one per employee, with their dept's avg)
df['dept_avg_salary']  = df.groupby('dept')['salary'].transform('mean')
df['dept_salary_rank'] = df.groupby('dept')['salary'].transform('rank')
df['above_dept_avg']   = df['salary'] > df['dept_avg_salary']

# Fill missing with GROUP median (the right way)
df['salary'] = df.groupby('dept')['salary'].transform(
    lambda x: x.fillna(x.median())
)
```

> **Used in Stage 1:** Creating contextual features like `price_vs_neighborhood_avg` for a house price model. Adding group context often gives a 5–15% accuracy boost.

## SECTION 9 — Merge and Join

---

### `pd.merge()` ⭐⭐⭐ 🧠 Pattern, 📖 Options

| | |
|---|---|
| **What** | Combine two DataFrames by matching rows on a shared column — exactly like SQL JOIN. |
| **Why** | Real ML data lives in multiple tables. You join features from different sources into one training dataset. |
| **When** | Building a feature table from multiple data sources. |
| **Where** | Data preparation — before training, after loading each source. |
| **Who** | Essential for anyone working with relational or structured data. |
| **How** | Specify the key column to match on and the join type. You already know SQL — this is identical. |
| **How often** | Any project with data from more than one table or file. |

```python
employees   = pd.DataFrame({'emp_id': [1,2,3,4], 'name': ['Alice','Bob','Carol','Dave'], 'dept_id': [10,20,10,30]})
departments = pd.DataFrame({'dept_id': [10,20,30], 'dept_name': ['Engineering','Sales','HR']})

# Inner — only rows with a match in BOTH (like SQL INNER JOIN)
pd.merge(employees, departments, on='dept_id', how='inner')

# Left — ALL employees, NaN if no matching dept (most common in ML)
pd.merge(employees, departments, on='dept_id', how='left')

# Outer — ALL rows from both, NaN where no match
pd.merge(employees, departments, on='dept_id', how='outer')
```

> **Most common in ML:** `left` join. You have your main dataset (left) and enrich it with features from another table (right). You always keep all rows from your main dataset.

> **Used in Stage 5:** Joining document metadata (title, source, date) with the embedding vectors before storing in a RAG vector database.

## SECTION 10 — Pivot and Reshape

---

### `df.pivot_table()` ⭐⭐ 📖

| | |
|---|---|
| **What** | Create a summary grid: rows = one category, columns = another category, values = an aggregation. |
| **Why** | Quickly reveal interactions between two categorical variables — e.g., "how does salary differ by dept AND seniority?" |
| **When** | EDA — understanding patterns across two dimensions before deciding what features to engineer. |
| **Where** | Analysis step. Also useful in Stage 7 for analyzing model performance across hyperparameter combinations. |

```python
df.pivot_table(
    values='salary',       # what to measure
    index='dept',          # rows
    columns='seniority',   # columns
    aggfunc='mean',        # how to aggregate
    fill_value=0           # replace NaN with 0
)
```

---

### `pd.melt()` — Wide → Long ⭐⭐ 📖

| | |
|---|---|
| **What** | Convert a wide table (many columns) into a long table (fewer columns, more rows). |
| **Why** | Many plotting libraries (seaborn) and some ML models expect "long format" data where each observation is one row. |
| **When** | When you have one column per time period or category and need to reshape for plotting or modeling. |

```python
df_wide = pd.DataFrame({'name': ['Alice','Bob'], 'salary': [90000,70000], 'bonus': [9000,7000]})
df_long = pd.melt(df_wide, id_vars=['name'], value_vars=['salary','bonus'],
                  var_name='metric', value_name='amount')
#    name   metric  amount
# 0  Alice  salary   90000
# 1  Bob    salary   70000
# 2  Alice  bonus     9000
# 3  Bob    bonus     7000
```

## SECTION 11 — String and DateTime Features

---

### `.str` Accessor ⭐⭐⭐ 🧠 Exists, 📖 Specific Methods

| | |
|---|---|
| **What** | Apply string operations to every value in a text column, vectorized. |
| **Why** | Real text data is messy — mixed case, extra spaces, inconsistent formats. You clean and extract features from it. |
| **When** | Text cleaning, extracting features from strings, normalizing categorical values. |
| **Where** | Preprocessing in Stage 1. NLP text cleaning in Stage 3. |

```python
df['name'].str.lower()                         # normalize case
df['name'].str.strip()                         # remove extra whitespace
df['name'].str.replace(' ', '_')               # replace characters
df['email'].str.split('@').str[1]              # extract domain
df['name'].str.contains('ali', case=False)     # filter by pattern
df['code'].str.extract(r'(\d{4})')             # regex extraction
df['name'].str.len()                           # string length as a feature
```

> **Used in Stage 3:** Cleaning raw text before tokenizing for an LLM or text classifier.

### `.dt` Accessor ⭐⭐⭐ 🧠 Exists, 📖 Specific Attributes

| | |
|---|---|
| **What** | Extract date/time components (month, hour, weekday) from a datetime column. |
| **Why** | Raw timestamps are useless to ML models. `month=3`, `hour=2`, `is_weekend=True` are meaningful numeric features. |
| **When** | Feature engineering on any dataset containing dates or timestamps. |
| **Where** | Stage 9 (time-series forecasting), fraud detection, user behavior modeling. |

```python
df['date'] = pd.to_datetime(df['date_str'])    # always convert strings first

# These become model input features
df['month']       = df['date'].dt.month
df['day_of_week'] = df['date'].dt.dayofweek    # 0=Monday, 6=Sunday
df['hour']        = df['date'].dt.hour
df['is_weekend']  = df['date'].dt.dayofweek >= 5
df['quarter']     = df['date'].dt.quarter
df['days_since']  = (pd.Timestamp.now() - df['date']).dt.days
```

> **Real scenario:** A fraud detection model doesn't care that a transaction happened on "2024-03-15 02:30:00". It cares that `hour=2` (2am) and `is_weekend=True`. Those engineered features catch fraud patterns the raw timestamp never could.

| **What** | Convert a cleaned DataFrame into a NumPy array — the required input format for any ML model. |

| **When** | Final step of every preprocessing pipeline — after all cleaning and feature engineering. |
| **How** | `.to_numpy()` or `.values` — both work, `.to_numpy()` is the modern preferred way. |

## Key Takeaways — Read This Before Any Project

1. **Always use a virtual environment** — one per project, never install AI packages globally
2. **Jupyter is for exploration** — run one cell, see output, tweak, rerun. Not for running full scripts
3. **VS Code + Jupyter extension** = best of both worlds — notebooks inside your editor
4. **Never commit `.venv/`** to Git — it's hundreds of MB of packages that can be recreated from `requirements.txt`
5. **Python 3.10+ required** — most AI libraries dropped support for older versions

## SECTION 1 — Python Virtual Environment

### `python -m venv` ⭐⭐⭐ 🧠

| | |
|---|---|
| **What** | Creates an isolated Python environment with its own packages — separate from your system Python. |
| **Why** | AI projects have strict dependency versions. NumPy 1.24 might break with TensorFlow 2.10. If you install everything globally, projects conflict. A venv gives each project its own isolated space. |
| **When** | First thing you do when starting any new AI/ML project. |
| **Where** | Local dev machine, CI/CD pipelines, Docker containers — same concept everywhere. |
| **Who** | Every Python developer. Non-negotiable in professional environments. |
| **How** | Python creates a `.venv` folder with its own Python interpreter and package registry. |
| **How often** | Once per project. |

```bash
# Create environment
python -m venv .venv

# Activate (Windows)
.venv\Scripts\activate

# Activate (Mac/Linux)
source .venv/bin/activate

# You'll see (.venv) in your terminal — means it's active
# All pip installs now go into this environment only

# Deactivate when done
deactivate
```

> **Analogy from your background:** A venv is exactly like a Docker container for Python packages. Each project gets its own isolated runtime. Same principle — different implementation.

> **Common mistake:** Installing packages without activating the venv first. They go into global Python and you'll wonder why `import numpy` fails inside your notebook.

## SECTION 2 — Installing the AI Stack

### `pip install` ⭐⭐⭐ 🧠

| | |
|---|---|
| **What** | Installs Python packages into your active virtual environment. |
| **Why** | All AI libraries (NumPy, Pandas, PyTorch, scikit-learn) are third-party — they don't come with Python. |
| **When** | After creating and activating a venv, before writing any code. |
| **How** | Downloads from PyPI (Python Package Index) and installs into `.venv`. |

```bash
# Core AI stack for Stage 0–1
pip install jupyter jupyterlab numpy pandas matplotlib seaborn scikit-learn

# Save installed packages so others (or future you) can recreate the environment
pip freeze > requirements.txt

# Restore from requirements.txt (on a new machine or after cloning a repo)
pip install -r requirements.txt
```

**Packages installed and why:**

| Package | Why you need it |
|---------|----------------|
| `numpy` | Array math — foundation of all ML |
| `pandas` | Data loading, cleaning, transformation |
| `matplotlib` | Basic plotting |
| `seaborn` | Statistical visualization |
| `scikit-learn` | Classical ML models (Stage 1) |
| `jupyterlab` | Interactive notebook environment |

## SECTION 3 — Jupyter Lab

### Why Jupyter Exists ⭐⭐⭐ 🧠

| | |
|---|---|
| **What** | A browser-based interactive coding environment where you write and run code in "cells". |
| **Why** | AI/ML work is exploratory — you load data, inspect it, clean it, try a model, evaluate, adjust. Each step takes time. Jupyter lets you run steps independently without re-running everything. |
| **When** | Learning, data exploration, prototyping, building visualizations. |
| **Where** | Stage 0–2 heavily. Later stages shift to `.py` scripts for production code. |
| **Who** | Every data scientist and ML engineer for exploratory work. |
| **How** | Each "cell" runs independently. The kernel holds state — variables defined in one cell are available in all cells below. |
| **How often** | Every day during learning and experimentation. |

```bash
# Launch
jupyter lab
# Opens at http://localhost:8888 in your browser
```

**Key shortcuts you must know:**

| Shortcut | Action |
|----------|--------|
| `Shift+Enter` | Run cell, move to next |
| `Ctrl+Enter` | Run cell, stay on same cell |
| `A` (command mode) | Insert cell above |
| `B` (command mode) | Insert cell below |
| `DD` (command mode) | Delete cell |
| `M` (command mode) | Convert cell to Markdown |
| `Esc` | Enter command mode |
| `Enter` | Enter edit mode |

> **Analogy:** Jupyter is like a REPL (you've used Python REPL) but it remembers every variable across all cells, lets you re-run individual steps, and shows output inline. Think of it as an interactive lab notebook.

> **Used in Stage 1:** You'll load the Titanic dataset in cell 1, inspect it in cell 2, clean in cell 3, train a model in cell 4, evaluate in cell 5 — without ever reloading the dataset.

## SECTION 4 — Project Folder Structure

### Standard AI Project Layout ⭐⭐⭐ 🧠

| | |
|---|---|
| **What** | A consistent folder structure for every AI project. |
| **Why** | As projects grow, unorganized files become unmanageable. A standard structure makes it easy to find data, code, models, and results. |
| **When** | Set up at the start of every project. |
| **Where** | Local dev, and mirrors your production repo structure. |

```
ai-learning/                  ← project root
├── .venv/                    ← isolated packages (NEVER commit to Git)
├── notebooks/                ← Jupyter notebooks (.ipynb files)
│   ├── 0.2-NumPy.ipynb
│   ├── 0.3-Pandas.ipynb
│   └── ...
├── data/                     ← raw and processed datasets
│   ├── raw/                  ← original data, never modified
│   └── processed/            ← cleaned data ready for models
├── src/                      ← reusable Python modules (.py files)
├── models/                   ← saved trained models
├── requirements.txt          ← package list for reproducibility
└── .gitignore                ← exclude .venv, data, model files from Git
```

> **Why separate `raw/` and `processed/`?** Raw data is sacred — never modify it. If your cleaning script has a bug, you want to re-run it from the original. This is a principle you'll see enforced in MLOps (Stage 7) with DVC (Data Version Control).

## SECTION 5 — VS Code Setup

### VS Code Extensions ⭐⭐ 🧠 Exists, 📖 Setup

| | |
|---|---|
| **What** | VS Code with Python and Jupyter extensions — edit scripts AND notebooks in one tool. |
| **Why** | As you progress from Stage 0 (notebooks) to Stage 1+ (scripts + notebooks), you need one environment for both. |
| **When** | Set up once. Use throughout entire learning plan. |

**Install these extensions in VS Code:**

| Extension | Publisher | Purpose |
|-----------|-----------|---------|
| Python | Microsoft | Python language support, linting, IntelliSense |
| Jupyter | Microsoft | Open and run `.ipynb` notebooks directly in VS Code |
| Pylance | Microsoft | Fast type checking and autocomplete |

**Running code in VS Code:**

| Action | How |
|--------|-----|
| Run a `.ipynb` notebook cell | `Shift+Enter` |
| Run entire `.py` script | `F5` or right-click → Run Python File |
| Run selected lines | Select code → `Shift+Enter` |
| Open terminal | `` Ctrl+` `` |
